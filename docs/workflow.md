# 存量埋点治理工作流程（严格遵循）

## 1. 目标与范围

### 1.1 流程目标

- 将输入的单条存量埋点补全为符合当前 BOSS 直聘埋点治理规范的基础属性结果。
- 在补全过程中保留推理依据，确保结果可解释、可复核。

### 1.2 处理范围

- 仅处理基础属性字段：
  - 页面位置
  - 功能名称
  - 事件类型
  - 埋点中文名
  - 埋点描述
  - 埋点英文名
- 仅使用以下信息源：
  - 输入埋点 JSON
  - 与该埋点绑定的需求文档

### 1.3 非目标（不处理）

- 不补充输入之外且无法从需求文档推导的业务事实。
- 不输出与治理规范无关的扩展字段（除非后续明确新增）。

---

## 2. 输入准备与受理

### 2.1 输入格式要求

- 输入为单条 JSON（key=字段名，value=字段值）。

```json
{
  "input_id": "TODO",
  "origin_fields": {
    "页面位置": "TODO",
    "功能名称": "TODO",
    "事件类型": "TODO",
    "埋点中文名": "TODO",
    "埋点描述": "TODO"
  }
}
```

- 需求文档可作为附加上下文输入。

### 2.2 受理前检查清单

- [ ] JSON 格式合法（可解析）
- [ ] 至少包含 1 个基础属性字段
- [ ] 若提供需求文档，文档与埋点存在明确绑定关系
- [ ] 输入中不存在明显乱码或不可识别字段名

---

## 3. 推理补全（核心阶段）

### 3.1 补全优先级（默认）

1. 根据输入 JSON 中各字段的已有值来推理出各字段规范化之后的值
2. 需求文档中的直接描述
3. 规范中的可推导规则
4. 历史样例（如后续接入）

### 3.2 字段级补全流程（逐字段执行，核心阶段中的核心步骤，你需要严格按照此流程来进行工作）

根据输入中埋点的历史信息（JSON格式的各字段已有的值）来推理得到此埋点各字段规范化后的值，工作流程如下：

1 先规范化埋点描述

- 如果原JSON中埋点描述字段存在值，并且内容较丰富，则不做更改
- 如果原JSON中埋点描述不存在值或者内容很少，就需要检索需求文档里与该埋点相关的内容来总结埋点描述。在检索文档之前你需要先对需求文档与埋点之间的相关性做判断，判断规则如下：
  - 先根据需求文档的名称来判断与埋点中已存在的字段值（比如中文名称、功能名称等字段的值）的语义关联性，如果关联性较强，就读取完整的文档内容并检索埋点相关的内容来总结埋点描述；如果语义关联性很弱，就不读取需求文档，不处理埋点描述这个字段。

2 规范化页面位置

- 如果原JSON中页面位置字段存在值，就保留原来的值。然后提取埋点描述中与页面位置有关的信息，到页面位置的枚举值集合中匹配出合适的枚举值补充进页面位置字段。如果埋点描述是空值，并且页面位置也是空值，就不做更改，仍然保持为空值。

3 规范化功能名称

- 如果原JSON中功能名称字段存在值，则将存在的值和从埋点描述中提取出来的与功能名称有关的信息进行融合再总结，得出标准的能够展示埋点业务作用和使用场景的总结内容作为功能名称。

- 【注意】功能名称中不要出现“点击、曝光、浏览等”事件类型的枚举值字眼。

4 规范化事件类型

- 如果原JSON中事件类型字段存在值，则不做处理，沿用原来的值作为规范之后的事件类型值；如果不存在，则从事件类型枚举集合（点击、曝光、浏览、时长、其他）中选择与埋点描述最相关的事件类型。

5 规范化中文名称

- 使用'-'拼接页面位置、功能名称、事件类型，得到"页面位置-功能名称-事件类型"这种格式的中文名称

---

## 4. 输出结构（标准产物）

### 4.1 补全结果 JSON（建议结构）

```json
{
  "input_id": "与传入的input_id保持一致",
  "status": "completed",
  "completed_fields": {
    "页面位置": "TODO",
    "功能名称": "TODO",
    "事件类型": "TODO",
    "埋点中文名": "TODO",
    "埋点描述": "TODO"
  }
}
```
