# 存量埋点治理工作流程（严格遵循）

## 1. 目标与范围

### 1.1 流程目标

- 将输入的单条存量埋点补全为符合当前 BOSS 直聘埋点治理规范的基础属性结果。
- 在补全过程中保留推理依据，确保结果可解释、可复核。

### 1.2 处理范围

- 仅处理基础属性字段：
  - 页面位置
  - 功能名称
  - 事件类型
  - 埋点中文名
  - 埋点描述
  - 埋点英文名
- 仅使用以下信息源：
  - 输入埋点 JSON
  - 与该埋点绑定的需求文档

### 1.3 非目标（不处理）

- 不补充输入之外且无法从需求文档推导的业务事实。
- 不输出与治理规范无关的扩展字段（除非后续明确新增）。

---

## 2. 输入准备与受理

### 2.1 输入格式要求

- 输入为单条 JSON（key=字段名，value=字段值）。

```json
{
  "input_id": "TODO",
  "origin_fields": {
    "页面位置": "TODO",
    "功能名称": "TODO",
    "事件类型": "TODO",
    "埋点中文名": "TODO",
    "埋点描述": "TODO"
  }
}
```

- 需求文档可作为附加上下文输入。

### 2.2 受理前检查清单

- JSON 格式合法（可解析）。
- 至少包含 1 个基础属性字段。
- 若提供需求文档，文档与埋点存在明确绑定关系。
- 输入中不存在明显乱码或不可识别字段名。

---

## 3. 推理补全（核心阶段）

### 3.1 补全优先级（默认）

1. 根据输入 JSON 中各字段已有值，推理规范化后的字段值。
2. 需求文档中的直接描述。
3. 规范中的可推导规则。
4. 历史样例（如后续接入）。

### 3.2 字段级补全流程（必须按顺序逐字段执行）

根据输入埋点的历史信息（JSON 中各字段已有值），推理得到该埋点各字段规范化后的值。执行顺序如下：

#### 1）先规范化埋点描述

- 若原 JSON 的 `埋点描述` 字段存在且内容较丰富，则不做更改。
- 若原 JSON 的 `埋点描述` 不存在或内容很少，则需要检索需求文档中与该埋点相关的内容进行总结。
- 在检索文档前，先判断需求文档与埋点的相关性，规则如下：
  - 先根据需求文档标题，判断其与埋点已有字段值（如中文名称、功能名称等）的语义关联性。
  - 若语义关联性强，则读取完整文档并检索相关内容，补全埋点描述。
  - 若语义关联性弱，则不读取需求文档，也不处理埋点描述字段。

#### 2）规范化页面位置

- 若原 JSON 的 `页面位置` 字段有值，则保留原值；原JSON中页面位置的格式可能是“/APP_C/职位列表/搜索”这种形式，你仍然需要去页面位置枚举文件进行匹配，拿到合适值进行补充。
- 同时可从埋点描述中提取页面位置信息，并在枚举集合中匹配合适值进行补充。
- 若多个枚举值都合适，可填写多个页面位置，使用英文逗号 `,` 分隔。
- 若 `埋点描述` 为空且 `页面位置` 也为空，则保持为空，不做更改。

【注意】

- 页面位置枚举文件为 `configs/enums/page_locations.json`。
- 页面位置必须来自枚举集合，不允许使用枚举外值。
- 枚举为树形结构（含多级路径），匹配时需满足以下约束：
  - 页面位置支持多个合法枚举值，使用英文逗号 `,` 连接。
  - 每个值必须是树形枚举中的完整层级路径（如 `APP_B-首善-发布职位`）。
  - 埋点中文名仍按 `cn_name-功能名称-事件类型` 拼接；当页面位置有多个值时，默认取逗号分隔后最末尾页面位置对应的 `cn_name` 参与拼接（中文名还有特殊规则，见第 5 步）。
  - 其余约束不变：只用枚举已有值、功能名称不含事件词、事件类型按枚举补全。
  - 若可确定为纯服务端埋点，页面位置直接取“无页面位置”，对应 `cn_name=server`。

#### 3）规范化功能名称

- 若原 JSON 的 `功能名称` 字段有值，则将其与从埋点描述中提取的功能信息融合总结，得到能体现业务作用和使用场景的标准功能名称。
- 功能名称中不要出现“点击、曝光、浏览”等事件类型枚举词。

#### 4）规范化事件类型

- 若原 JSON 的 `事件类型` 字段有值，则沿用原值，不做处理。
- 若无值，则从事件类型枚举集合中选择与埋点描述最相关的值：`点击、曝光、浏览、时长、其他`。

#### 5）规范化中文名称

- 使用 `-` 拼接页面位置对应的 `cn_name`、功能名称、事件类型，形成 `cn_name-功能名称-事件类型` 格式。
- 若页面位置存在多个值，默认取最末尾页面位置的 `cn_name` 用于中文名拼接。
- 若多个页面位置有相同上级目录（例如 `APP_B-首善-企业认证,APP_B-首善-实名认证`），且末尾路径业务含义不同，则中文名中的页面位置部分合并为共同上级对应的 `cn_name`（此例为 `B_首善`）。
- 若多个页面位置有相同上级目录（例如 `APP_C-我的-其他功能-我的客服,小程序_C-我的-其他功能-我的客服,APP_B-我的-其他功能-我的客服`），且末尾路径业务含义相同，则中文名页面位置部分保留末尾路径不变（此例为 `BC_我的_其他功能_我的客服`）。
- 若页面位置为多值且同时包含 B 端与 C 端，则中文名称中页面位置一级路径改写为 `BC`，避免信息遗漏。

---

## 4. 输出结构（标准产物）

### 4.1 补全结果 JSON（建议结构）

```json
{
  "input_id": "与传入的input_id保持一致",
  "status": "completed",
  "completed_fields": {
    "页面位置": "TODO",
    "功能名称": "TODO",
    "事件类型": "TODO",
    "埋点中文名": "TODO",
    "埋点描述": "TODO"
  }
}
```
